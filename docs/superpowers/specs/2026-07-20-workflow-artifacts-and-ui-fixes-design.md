# 워크플로우 자료 연동 및 관리자 UI 개선 설계

## 목표

다음 네 가지 문제를 하나의 구현 단위로 해결한다.

1. 관리자 대시보드의 올리비아 비서가 좁은 오른쪽 열에서 펼쳐질 때 잘리는 문제를 해결한다.
2. 견적서, 계약서, 콘티의 생성 당시 PDF 원본을 보존하고 고객 진행페이지에서 확인·다운로드할 수 있게 한다.
3. 견적서 PDF에 안전 여백을 확보하고 하단 대표 사인이 잘리지 않게 한다.
4. 콘티 생성 화면의 전체 배경을 옅은 그린 계열로 통일한다.

캘린더 일정별 알람과 올리비아 텔레그램 발송은 이 작업이 끝난 뒤 별도 2단계로 설계·구현한다.

## 대시보드 레이아웃

관리자 홈의 상단 순서를 다음과 같이 구성한다.

1. 브리핑, 오늘 스케줄, 오늘의 명언 3개 카드
2. 전체 폭 올리비아 비서
3. KPI와 운영 알림
4. 진행 현황과 보조 패널

올리비아 비서는 접힌 상태에서 낮은 높이를 유지한다. 펼친 상태에서는 페이지의 전체 가로 폭을 사용해 탭, 카드, 문장이 오른쪽 화면 경계에서 잘리지 않게 한다. 올리비아 추천 액션과 최근 활동은 기존 보조 패널에 유지한다.

반응형 화면에서는 상단 3개 카드가 기존 규칙에 따라 한 열로 전환되고, 올리비아 비서도 그 아래에서 한 열 전체를 사용한다.

## 공통 자료 저장 구조

### Storage

비공개 Supabase Storage 버킷 `workflow-artifacts`를 사용한다. 저장 경로는 다음 규칙을 따른다.

```text
{client_id}/{workflow_run_id-or-unassigned}/{document_type}/{artifact_id}/{safe_file_name}
```

원본 파일은 공개 URL로 노출하지 않는다. 보기 또는 다운로드 요청 시 서버가 짧은 유효기간의 서명 URL을 발급한다.

### 데이터베이스

공통 메타데이터 테이블 `workflow_artifacts`를 추가한다.

- `id`: 자료 UUID
- `client_id`: 연결된 고객
- `workflow_run_id`: 연결된 워크플로우 실행, 연결할 실행이 없으면 null
- `workflow_step_key`: 생성 단계 (`quote`, `contract`, `conti`)
- `document_type`: `quote`, `contract`, `conti`
- `source_table`: 원본 데이터 테이블
- `source_id`: 견적서·계약서·콘티 원본 레코드 ID
- `title`: 고객 화면에 표시할 제목
- `file_name`: 생성 당시 파일명
- `storage_path`: 비공개 버킷 내부 경로
- `mime_type`: 초기값 `application/pdf`
- `file_size`: 바이트 단위 크기
- `status`: `ready` 또는 `upload_failed`
- `created_at`, `updated_at`: 생성·수정 시각

`client_id`, `workflow_run_id`, `document_type`, `created_at` 조회를 위한 인덱스를 둔다. 같은 원본 레코드의 동일 파일이 중복 등록되지 않도록 소스와 파일 식별자를 기준으로 제약을 둔다. 테이블과 버킷은 서버의 service role을 통해서만 변경한다.

견적서, 계약서, 콘티 원본 테이블에는 누락된 `client_id`와 `workflow_run_id` 연결 컬럼을 additive migration으로 보강한다. 기존 설치 환경에서 반복 실행해도 안전한 SQL로 작성한다.

## 문서 생성과 업로드 흐름

세 문서 모두 다음 공통 흐름을 따른다.

1. 문서 데이터를 원본 DB 테이블에 저장하고 원본 레코드 ID를 확보한다.
2. 화면에 표시된 최종 내용으로 PDF Blob을 생성한다.
3. PDF와 고객·워크플로우·원본 레코드 메타데이터를 서버 업로드 API로 전송한다.
4. 서버가 파일을 비공개 Storage에 저장한 뒤 `workflow_artifacts`를 `ready` 상태로 기록한다.
5. 브라우저는 Storage에 보관된 것과 동일한 Blob을 다운로드한다.

견적서와 계약서는 현재 jsPDF 생성 함수를 Blob 반환 방식으로 정리해 저장과 다운로드가 같은 파일을 사용하게 한다.

콘티는 현재 브라우저 인쇄창이 파일 바이트를 앱에 돌려주지 않으므로, 인쇄 전용 HTML을 캡처해 다중 페이지 PDF Blob을 만드는 방식으로 전환한다. 한글은 이미지 기반으로 포함해 폰트 임베딩 문제를 피한다. PDF 생성 결과가 준비된 뒤 업로드와 다운로드를 수행한다.

고객이나 워크플로우 실행 ID가 URL에 전달된 경우 이를 우선 사용한다. 없는 경우 병원명으로 고객을 확인하고 해당 고객의 최신 활성 워크플로우 실행에 연결한다. 명확한 워크플로우가 없으면 고객에는 연결하되 `workflow_run_id`는 null로 남긴다.

## 고객 진행페이지 관련 자료

고객 상세 API는 선택된 워크플로우 실행에 연결된 `ready` 자료와 고객 공통 자료를 함께 반환한다. 진행페이지에는 기존의 분리된 단순 견적서·계약서 목록 대신 `관련 자료` 섹션을 둔다.

각 행에는 다음 정보를 표시한다.

- 문서 유형 배지
- 제목과 원본 파일명
- 생성일과 파일 크기
- `보기` 버튼
- `다운로드` 버튼

`보기`는 새 탭에서 PDF를 열고, `다운로드`는 원래 파일명으로 저장한다. 두 동작 모두 자료 ID 기반 서버 API를 거쳐 서명 URL을 발급받는다. Storage 경로를 클라이언트 요청값으로 직접 받지 않는다.

기존 DB 레코드에 원본 파일이 없는 경우 메타데이터 목록은 유지하되 `원본 파일 없음`으로 표시한다. 이를 원본이 존재하는 자료처럼 다운로드 가능하게 만들거나 현재 데이터로 재생성하지 않는다.

## 견적서 PDF 레이아웃

참고 PDF는 A4 가로 캔버스를 끝까지 사용하고, 하단 서명 영역이 캡처 높이 밖으로 내려가 대표 사인이 잘린다. 다음을 함께 조정한다.

- A4 출력 이미지에 안전 여백을 둔다.
- 캡처용 견적서의 좌우·상하 내부 여백과 하단 3열 간격을 재조정한다.
- 계약 안내 문구와 대표 서명 영역이 같은 페이지 높이 안에 들어오도록 하단 영역을 압축한다.
- 서명 이미지와 라벨에 최소 하단 여백을 확보한다.
- 견적 항목이 늘어날 때 무조건 잘라내지 않고, 지원 범위를 넘으면 명확한 생성 오류를 표시한다.

최종 PDF는 A4 가로 한 페이지를 유지한다. 캡처 이미지 전체를 페이지 경계에 딱 맞추지 않고 안전 영역 안에 배치한다.

## 콘티 배경

콘티 전용 레이아웃의 기본색과 그라데이션을 옅은 그린 계열로 변경한다. 사진 콘티 페이지 내부의 불투명한 아이보리 최상위 배경도 같은 계열로 바꿔 레이아웃 배경을 덮지 않게 한다. 카드와 표는 흰색을 유지해 대비와 가독성을 보존한다.

## 오류 처리

- 원본 DB 저장 실패: PDF 생성과 업로드를 시작하지 않고 저장 실패를 표시한다.
- PDF 생성 실패: DB 데이터는 유지하고 원본 파일 생성 실패와 재시도 방법을 표시한다.
- Storage 업로드 실패: `ready` 자료를 만들지 않으며 사용자에게 업로드 재시도를 제공한다.
- 메타데이터 저장 실패: 업로드한 객체를 정리하고 실패를 반환해 고아 파일 생성을 막는다.
- 보기·다운로드 URL 발급 실패: 만료 또는 권한 오류를 구분하지 않고 안전한 사용자 메시지를 표시한다.
- 선택한 워크플로우와 자료의 고객이 일치하지 않으면 서버가 요청을 거부한다.

## 성능과 보안

- 세 문서가 공유하는 업로드·자료 타입과 클라이언트 호출 함수를 공통 모듈로 둔다.
- 독립적인 고객 상세 조회는 병렬로 수행한다.
- PDF 생성용 라이브러리는 기존처럼 사용자 작업 시 동적으로 불러와 초기 번들에 포함하지 않는다.
- 파일 크기와 MIME 타입을 서버에서 검증한다.
- 파일 조회는 자료 ID만 허용하고 Storage 경로를 직접 노출하지 않는다.

## 검증

1. TypeScript 검사와 기존 자동 테스트를 실행한다.
2. additive migration과 API의 필수 입력, 고객·워크플로우 연결 검증을 확인한다.
3. 실제 브라우저에서 대시보드 올리비아 비서를 펼쳐 데스크톱·모바일 잘림 여부를 확인한다.
4. 견적서·계약서·콘티를 각각 저장하고 동일 원본이 고객 진행페이지에 표시되는지 확인한다.
5. `보기`와 `다운로드`가 작동하며 파일명이 유지되는지 확인한다.
6. 견적서 PDF를 PNG로 렌더링해 바깥 여백, 표, 합계, 계약 안내, 로고, 대표 사인이 모두 잘리지 않는지 검사한다.
7. 콘티 페이지의 최상위와 콘텐츠 영역이 모두 옅은 그린으로 보이는지 확인한다.
8. 업로드 실패를 모의해 불완전한 자료가 정상 자료로 노출되지 않는지 확인한다.

## 후속 작업

이 구현과 검증이 끝난 뒤 캘린더 기능을 별도 작업으로 진행한다. 일정별 알람 활성화, 알림 시점, 올리비아 알림 이벤트, 텔레그램 메시지 발송과 중복 방지를 별도 설계 범위로 삼는다.
