# Olivia 대화형 업무 비서 2차 설계

- 작성일: 2026-07-18
- 대상: Olivia Agent 최신 `main`
- 선행 기능: Olivia Observer, 인사이트, 행동, 승인, 약속, 브리핑
- 목표: 기존 올리비아 채팅을 Observer·CRM·승인함과 연결한다.

## 1. 범위

이번 단계는 올리비아가 일반적인 질문에 답하는 수준을 넘어, 현재 운영 데이터를 조회하고 준비된 업무를 승인 흐름으로 연결하는 대화형 비서가 되도록 확장한다.

지원할 대표 대화는 다음과 같다.

- 오늘 긴급한 일과 우선순위 조회
- 고객 또는 프로젝트 진행 상태 조회
- 승인 대기, 대표·고객 약속, 최근 고객 반응 조회
- 고객 후속 메시지와 메일 초안 준비
- 준비된 Olivia 행동의 승인, 실행, 무시
- Observer 즉시 재점검
- 직전 조회 결과를 이용한 후속 표현 처리

기존 일반 대화, 이미지 분석, 캘린더, 견적, 계약, 콘티 및 페이지 이동 도구는 유지한다.

## 2. 선택한 접근

기존 `app/api/olivia/route.ts`에 데이터베이스 로직을 계속 추가하지 않는다. 채팅 API는 대화와 모델 호출을 담당하고, 신규 업무 도구 계층이 Observer·CRM·승인 데이터를 조회하고 기존 실행 계층을 호출한다.

```text
OliviaChat
  -> /api/olivia
    -> 대화 의도와 도구 선택
      -> Olivia 업무 도구 계층
        -> Observer / CRM / agent_tasks / agent_approvals
        -> 결과 정규화
    -> 텍스트 또는 구조화 업무 카드 응답
```

이 구조는 기존 채팅을 보존하면서 데이터 조회, 권한 판단, 행동 실행을 독립적으로 테스트할 수 있게 한다.

## 3. 구성 요소

### 3.1 업무 조회 서비스

신규 서버 모듈은 다음 책임을 가진다.

- 오늘 브리핑, 긴급 인사이트, 고객 반응 조회
- 고객명 또는 프로젝트명으로 활성 워크플로우 검색
- 프로젝트의 현재 단계, 다음 행동, 인사이트, 약속, 승인 요약
- 승인 대기와 약속 목록 조회
- 조회 결과를 채팅 카드용 공통 결과로 정규화

선택 테이블 또는 연결 컬럼이 없는 환경에서는 전체 요청을 실패시키지 않고 해당 항목만 빈 배열로 반환한다.

### 3.2 업무 실행 서비스

업무 실행 서비스는 신규 실행 로직을 복제하지 않고 다음 기존 기능을 재사용한다.

- `runOliviaObserver()`
- `planOliviaAction()` 및 `runOliviaAction()`
- 기존 `agent_approvals`
- 기존 `agent_tasks`
- 기존 `mailing_queue`
- 기존 workflow 실행·전환 함수

### 3.3 채팅 결과 문맥

조회 도구 결과는 안정적인 참조 ID와 순서를 포함한다. 최근 결과의 최소 문맥을 다음 요청에 전달하여 아래 표현을 해석한다.

- 첫 번째 항목
- 그 고객
- 방금 본 계약서
- 두 번째 승인

서버는 참조 ID를 다시 조회하고 현재 상태를 검증한 뒤 실행한다. 화면에서 전달된 제목이나 순번만 신뢰하지 않는다.

## 4. 채팅 도구

### 조회 도구

- `get_today_briefing`: 긴급, 승인, 약속, 고객 반응과 오늘 행동 조회
- `get_urgent_insights`: 열린 긴급 인사이트 조회
- `search_client_projects`: 고객명 또는 프로젝트명 검색
- `get_project_status`: 현재 단계, 다음 행동, 문제, 최근 반응 조회
- `list_pending_approvals`: 기존 승인과 Olivia 승인 대기 조회
- `list_commitments`: 대표·고객 약속, 임박, 기한 초과 조회

### 준비·실행 도구

- `prepare_followup`: 고객 후속 메시지 또는 메일 초안 준비
- `manage_olivia_action`: Olivia 행동의 준비, 승인, 실행, 무시
- `run_observer`: 활성 업무 또는 지정 워크플로우 즉시 점검

도구 이름과 입력 스키마는 서버에서 고정하며 모델이 임의의 테이블명, SQL 또는 실행 API를 선택하게 하지 않는다.

## 5. 권한과 승인

### 즉시 실행

- 브리핑, 인사이트, 프로젝트, 승인, 약속 조회
- Observer 재점검
- 내부 초안 준비
- 내부 상태 요약

### 승인 카드 필요

- 고객 이메일과 후속 연락 발송
- 고객 포털 공개
- 견적·계약·콘티 공개
- 워크플로우 단계 이동
- 외부 공유와 직원 배정

### 대표 직접 확인

- 가격과 할인 변경
- 계약 조건 변경
- 결제와 환불
- 파일 삭제
- 고객 불만 답변
- 외부 콘텐츠 게시
- 개인정보 외부 전송

“승인해줘”의 대상이 한 건으로 확정될 때만 승인 단계를 진행한다. 후보가 여러 건이면 선택 목록을 다시 보여준다. 승인 직전에는 DB에서 대상 상태와 권한을 다시 조회한다.

## 6. 응답과 UI

텍스트 설명과 함께 선택적으로 구조화된 `workItems`를 반환한다. 각 항목에는 다음 정보가 포함된다.

```typescript
type OliviaChatWorkItem = {
  id: string;
  kind: "insight" | "action" | "approval" | "commitment" | "project" | "event";
  title: string;
  summary: string;
  clientName?: string;
  projectName?: string;
  priorityScore?: number;
  dueAt?: string;
  status?: string;
  reason?: string;
  availableActions: Array<"view" | "prepare" | "approve" | "run" | "snooze" | "dismiss">;
};
```

채팅은 기존 도구 승인 카드 스타일을 재사용한다. 업무 카드에는 긴급도, 고객·프로젝트, 기한, 중요한 이유와 가능한 행동을 표시한다. 모바일 버튼은 내용 길이에 따라 한 열 또는 두 열로 배치한다.

홈의 Olivia 업무 영역과 채팅은 동일한 API와 데이터 모델을 사용한다. 별도 복제 상태를 만들지 않는다.

## 7. 데이터와 기록

- 사용자·비서 메시지는 기존 `olivia_chat_messages`에 저장한다.
- 업무 조회 결과 전문은 메시지 본문에 중복 저장하지 않고 참조 ID와 축약 정보만 기록한다.
- 승인, 실행, 실패는 `olivia_events`에 기록한다.
- 행동 상태는 `olivia_actions`와 기존 승인·업무 테이블을 기준으로 한다.
- 이메일, 전화번호, 계약 전문, 원본 녹취는 채팅 실행 로그에 저장하지 않는다.

새로고침 시 서버 대화 기록을 우선 복구하되, 테이블을 사용할 수 없는 환경에서는 기존 로컬 저장 fallback을 유지한다.

## 8. 오류와 중복 처리

- 일부 조회가 실패해도 정상 조회된 항목은 반환한다.
- AI 호출이 실패해도 결정 가능한 조회 결과는 규칙 기반 텍스트와 카드로 반환한다.
- 승인·실행 실패는 완료로 표시하지 않고 오류와 재시도 가능 여부를 반환한다.
- 완료 또는 실행 중인 행동은 재실행하지 않는다.
- 동일한 후속 초안은 기존 행동의 deduplication key를 사용한다.
- 기존 일반 채팅은 Observer 또는 CRM 장애와 분리해 계속 동작한다.

## 9. 성능

- 서로 독립적인 조회는 `Promise.allSettled()`로 병렬 실행한다.
- 기본 조회 건수에 상한을 두고 상세 데이터는 사용자가 선택한 항목에 한해 가져온다.
- 클라이언트에는 카드 표시와 후속 선택에 필요한 최소 데이터만 전달한다.
- 무거운 업무 모듈은 서버에서만 import한다.

## 10. 테스트와 완료 기준

### 단위 테스트

- 오늘 업무 결과의 우선순위 정렬
- 고객·프로젝트 검색과 동명이인 후보 처리
- 최근 결과의 첫 번째·두 번째 참조 해석
- 승인 전 실행 차단
- 상태 재검증과 중복 실행 차단
- 부분 조회 실패 fallback

### 통합 테스트

1. 오늘 긴급 항목 조회 후 첫 항목의 고객 후속 초안 준비
2. 프로젝트 검색 후 현재 단계와 다음 행동 조회
3. 승인 대기 조회 후 단일 대상을 승인하고 실행
4. 여러 승인 후보에서 모호한 승인 요청 차단
5. Observer 재점검 후 신규 인사이트를 같은 대화에서 조회
6. DB 대화 기록 실패 시 로컬 대화 fallback 유지

### 완료 기준

- 채팅에서 Observer·CRM·승인·약속 데이터를 조회할 수 있다.
- 직전 업무 결과를 참조한 후속 대화가 가능하다.
- 외부 상태 변경은 승인 없이 실행되지 않는다.
- 실행 결과와 실패가 사건 로그에 기록된다.
- 기존 채팅 도구와 일반 대화가 정상 작동한다.
- 모바일·데스크톱 업무 카드가 기존 디자인과 일치한다.
- TypeScript 검사, 테스트, 프로덕션 빌드가 성공한다.

## 11. 제외 범위

- 완전 자율 다단계 Agent Loop
- 고객 메시지 자동 발송
- 가격·계약 조건 자동 변경
- 콘텐츠 자동 게시
- 모델 재학습
- 기존 CRM 또는 Observer 전면 재작성
