# Olivia 미팅 비서 채팅 연동 설계

- 작성일: 2026-07-18
- 대상: Olivia Agent 최신 `main`
- 선행 기능: Olivia Observer, 대화형 업무 비서, 캘린더, 메모·음성메모, CRM 워크플로우
- 목표: 고객 미팅 전 준비와 미팅 후 후속 업무를 채팅·홈·CRM에서 하나의 흐름으로 연결한다.

## 1. 범위

이번 단계는 캘린더의 고객 미팅을 감지하고, 고객·프로젝트 문맥을 이용한 미팅 전 브리핑과 메모 기반 미팅 후 분석을 제공한다.

지원할 대표 대화는 다음과 같다.

- 오늘 또는 내일 고객 미팅 조회
- 고객명으로 미팅 전 브리핑 생성
- 캘린더 일정과 고객·프로젝트 연결
- 일반메모·음성메모를 지정해 미팅 후 분석
- 대표·고객 약속 저장과 마감 관리
- 다음 행동 추천과 내부 업무 생성
- 고객 후속 연락과 견적 초안을 승인함에 준비

녹음 자동 수집, 고객 메시지 자동 발송, 가격·계약 조건 자동 변경은 제외한다.

## 2. 선택한 접근

채팅 요청형과 캘린더 능동 감지를 결합한다.

```text
calendar_tasks 고객 미팅
  -> 미팅 감지·연결 서비스
    -> 고객·워크플로우 후보 확인
      -> 미팅 전 브리핑 생성
        -> 채팅 업무 카드 / 관리자 홈

상담·음성 메모
  -> 미팅 후 분석 서비스
    -> 요구사항·확정·미확정·약속 추출
      -> next_action 추천
      -> 내부 업무 / 승인 필요 후속 행동
```

기존 `pre-brief`, `post-analysis` API와 `meeting_commitments`를 재사용한다. 채팅 API에 DB 로직을 직접 추가하지 않고 미팅 서비스 계층을 둔다.

## 3. 미팅 식별과 고객 연결

### 3.1 캘린더 미팅 판정

다음 조건 중 하나를 충족하면 고객 미팅 후보로 본다.

- `calendar_tasks.category = 'client'`
- 제목 또는 메모에 미팅·상담·회의가 포함되고 고객명이 식별됨
- 일정 메타데이터에 `clientId` 또는 `workflowRunId`가 존재함

단순 개인 일정이나 내부 회의는 고객 미팅으로 분류하지 않는다.

### 3.2 연결 우선순위

1. 일정에 저장된 `workflowRunId`
2. 일정에 저장된 `clientId`와 활성 워크플로우
3. 일정 제목·메모와 `workflow_runs.client_name`의 정확한 일치
4. 부분 일치 후보를 사용자에게 표시

후보가 여러 건이면 자동 연결하지 않는다. 고객·프로젝트 선택 카드를 반환한다.

### 3.3 additive metadata

기존 캘린더 테이블을 재작성하지 않는다. 필요한 연결 정보는 신규 `meeting_context jsonb` 컬럼에 저장한다.

```json
{
  "clientId": "uuid",
  "workflowRunId": "uuid",
  "meetingStatus": "scheduled",
  "briefingId": "uuid",
  "memoId": "uuid"
}
```

마이그레이션 전 환경에서는 제목·날짜 기반 조회 fallback을 유지한다.

## 4. 미팅 전 브리핑

브리핑은 다음 근거를 조합한다.

- 고객 기본 정보
- 최근 상담 메모 요약
- 현재 워크플로우 단계와 다음 행동
- 최근 견적·계약 상태
- 열린 승인
- 대표·고객 미완료 약속
- 이전 미팅의 미해결 질문
- 촬영 일정과 준비 누락

결과 섹션:

1. 고객과 프로젝트 요약
2. 현재 진행 상태
3. 이전 논의와 미해결 항목
4. 대표·고객 약속
5. 이번 미팅에서 반드시 확인할 질문
6. 주의할 고객 우려
7. 추천 서비스 또는 다음 행동

근거가 없는 값은 생성하지 않고 `확인 필요`로 표시한다.

### 중복 방지

동일 일정·워크플로우·날짜의 브리핑은 새 행을 계속 만들지 않는다. 기존 브리핑을 갱신하고 생성 시각만 업데이트한다.

```text
meeting_pre:{calendarTaskId}:{workflowRunId}:{meetingDate}
```

## 5. 미팅 후 분석

입력은 기존 상담 메모 또는 음성메모 분석 결과다. 원본 전체 녹취 대신 분석에 필요한 메모 본문만 서버에서 사용한다.

구조화 결과:

```typescript
type MeetingPostAnalysis = {
  customerNeeds: string[];
  confirmedItems: string[];
  unresolvedItems: string[];
  representativeCommitments: Array<{ text: string; dueAt?: string }>;
  clientCommitments: Array<{ text: string; dueAt?: string }>;
  objections: string[];
  budget?: string;
  desiredSchedule?: string;
  decisionMaker?: string;
  recommendedPackage?: string;
  nextAction?: string;
  confidence: number;
};
```

분석 후 처리:

- 약속을 `meeting_commitments`에 중복 제거 저장
- `consultation_memos.next_action` 유지
- `workflow_runs.next_action` 추천
- 필요한 내부 `agent_tasks` 생성
- 고객 확인 메시지 또는 견적 초안을 `olivia_actions`로 준비
- `meeting_post` 브리핑 저장
- CRM 타임라인용 `meeting.analyzed` 이벤트 기록

### 수동 다음 행동 보호

`workflow_runs.next_action_source = 'manual'`이고 값이 있으면 AI가 덮어쓰지 않는다. AI 추천은 분석 결과와 Olivia 행동에만 보관한다.

## 6. 채팅 도구

신규 도구:

- `list_upcoming_meetings`: 오늘·내일 또는 지정 기간의 고객 미팅 후보 조회
- `link_meeting_client`: 일정과 고객·워크플로우 연결
- `prepare_meeting_brief`: 미팅 전 브리핑 생성 또는 갱신
- `analyze_meeting_memo`: 지정 메모의 미팅 후 분석
- `complete_meeting`: 일정 상태를 완료로 변경하고 분석할 메모 선택 안내
- `get_meeting_followups`: 약속·내부 업무·승인 필요 후속 행동 조회

조회와 내부 브리핑 생성은 즉시 실행한다. 고객 연결 변경, 미팅 완료와 분석할 메모 선택은 사용자의 명시적 요청 또는 카드 버튼으로 시작한다. 사용자가 메모를 선택한 뒤의 구조화 분석·약속 저장은 별도 승인 없이 내부 작업으로 실행한다.

## 7. UI

홈과 채팅은 동일한 미팅 업무 카드 타입을 사용한다.

카드 표시:

- 고객·프로젝트명
- 미팅 일시와 장소
- 연결 상태와 준비 상태
- 현재 워크플로우 단계
- 확인할 질문 수
- 미완료 약속 수
- 중요 주의사항

행동:

- 브리핑 보기
- 고객·프로젝트 연결
- 메모 연결
- 미팅 완료
- 후속 업무 확인

미팅 후 분석 카드는 고객 요구사항, 확정·미확정 항목, 대표·고객 약속, 이의사항, 예산·일정, 추천 패키지, 다음 행동과 준비된 후속 업무를 구분해 표시한다.

모바일에서는 핵심 요약을 먼저 표시하고 상세 섹션은 펼침 형태로 제공한다.

## 8. 권한

### 자동 또는 즉시 실행

- 미팅 후보 조회
- 고객 문맥 조회
- 미팅 전 브리핑 생성·갱신
- 메모 구조화 분석
- 약속 중복 제거 저장
- 내부 다음 행동 추천

### 사용자 명시 동작 필요

- 일정과 고객·프로젝트 연결
- 미팅 완료 처리
- 분석할 메모 선택

### 대표 승인 필요

- 고객 후속 메시지 발송
- 견적·계약·콘티 공개
- 워크플로우 단계 이동
- 가격·할인·계약 조건 변경

## 9. 오류 처리

- 고객 연결이 없으면 브리핑을 추측하지 않고 연결 후보를 표시한다.
- 일부 문맥 조회가 실패하면 조회 가능한 정보로 규칙 기반 브리핑을 만든다.
- AI 분석 실패 시 원본 메모를 보존하고 재시도 가능한 실패 상태를 기록한다.
- 약속은 메모 ID, 담당자 유형, 정규화한 내용으로 중복을 방지한다.
- 일정 완료와 메모 분석은 독립적으로 처리해 한쪽 실패가 다른 데이터를 되돌리지 않게 한다.
- 개인정보와 전체 녹취를 `olivia_events` payload에 기록하지 않는다.

## 10. 테스트와 완료 기준

### 단위 테스트

- 오늘·내일 기간 계산과 한국 시간 처리
- 고객 미팅 판정
- 정확 일치·부분 일치·동명이인 후보 처리
- 브리핑 중복 키 생성
- 약속 중복 제거
- 수동 다음 행동 보호
- 부분 조회 실패 fallback

### 통합 테스트

1. 내일 고객 미팅 조회 후 단일 프로젝트 연결과 브리핑 생성
2. 여러 프로젝트 후보에서 자동 연결 차단
3. 미팅 메모 분석 후 대표·고객 약속 저장
4. 수동 다음 행동을 유지하면서 AI 추천 별도 저장
5. 고객 후속 메시지 초안을 승인함에 등록하고 자동 발송 차단
6. 동일 미팅 브리핑 재생성 시 기존 브리핑 갱신

### 완료 기준

- 채팅에서 오늘·내일 고객 미팅을 조회할 수 있다.
- 연결된 미팅의 근거 기반 사전 브리핑을 생성한다.
- 메모에서 약속과 다음 행동을 구조화한다.
- 후속 업무가 Observer·CRM·승인함에 연결된다.
- 고객 연락은 승인 없이 발송되지 않는다.
- 홈과 채팅에서 동일한 미팅 정보를 보여준다.
- TypeScript, 테스트, 프로덕션 빌드가 성공한다.

## 11. 제외 범위

- 마이크 상시 녹음과 자동 녹음 수집
- 실시간 회의 자막
- 고객 메시지 자동 발송
- 가격·계약 조건 자동 변경
- CRM 및 캘린더 전면 재작성
- 모델 재학습
