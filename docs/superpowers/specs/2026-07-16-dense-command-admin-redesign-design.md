# Olivia Admin Dense Command 리디자인 명세

## 1. 목적

Olivia Admin의 관리자 홈, 고객관리 CRM, 개별 기능 화면을 하나의 고밀도 운영 디자인으로 통합한다. 대표가 화면을 탐색하는 시간을 줄이고, 첫 화면에서 상태와 다음 행동을 빠르게 판단할 수 있게 하는 것이 목표다.

이번 작업은 UI 리디자인이다. 기존 기능의 생성·분류·매칭·메일·CRM 로직과 라우트 동작은 변경하지 않는다.

## 2. 승인된 디자인 방향

선택된 방향은 `Dense Command`다.

- 밝은 작업 공간과 딥 티얼 사이드바를 결합한 관제탑형 UI
- 넓은 장식 영역보다 정보 탐색 속도와 상태 비교를 우선
- 포토클리닉 딥 티얼을 기본색으로 사용하고 오렌지는 경고와 핵심 실행에 제한
- 관리자 홈, CRM, 개별 기능이 같은 헤더·사이드바·카드·상태 체계를 공유
- 조밀하지만 카드 간격과 내부 여백을 분리해 답답하지 않게 구성

## 3. 범위

### 포함

- 공통 `AdminShell`, `AdminSidebar`, `AdminHeader` 시각 개편
- `/admin/dashboard/home` 관리자 홈 개편
- `/admin/crm/dashboard` CRM 대시보드 개편
- `/admin/tools` 개별 기능 홈 개편
- 요약 카드, 업무 카드, 보드, 상태 배지, 추천 패널의 공통 디자인 통일
- 개별 기능 카드의 1:1 스퀘어 그리드 적용
- 데스크톱과 모바일 반응형 재구성
- 정상, 로딩, 빈 데이터, 연결 오류 상태의 시각 구분
- 키보드 포커스, 축소 모션, 상태 알림 접근성 보완

### 제외

- 메뉴 항목 추가·삭제 또는 카테고리 재분류
- `lib/toolNav.ts`와 `/admin/tools`의 기능 목록 통합
- CRM 데이터 모델 및 워크플로우 로직 변경
- 견적서, 계약서, 콘티, 사진 분류, 셀렉, RAW 매칭 등 실제 기능 로직 변경
- Supabase 스키마 또는 API 응답 구조 변경
- 메일 및 자동화 로직 변경

현재 사이드바와 `/admin/tools`가 서로 다른 기능 목록을 사용하는 구조는 이번 디자인 작업에서 유지한다. 메뉴 데이터 정규화는 별도 기능 작업으로 다룬다.

## 4. 공통 레이아웃

### 데스크톱

- 사이드바: 224px 고정 폭
- 헤더: 72px 높이
- 콘텐츠: 최대 폭을 강제로 좁히지 않고 가용 영역 전체 사용
- 기본 그리드: 12열
- 카드 간격: 14~18px
- 페이지 좌우 여백: 20~24px
- 메인 콘텐츠와 보조 패널: 기본 8:4 비율

### 모바일

- 사이드바는 드로어로 전환하고 닫힌 상태에서 `inert`, `aria-hidden`, `visibility: hidden` 적용
- 헤더는 제목과 필수 액션만 유지
- 관리자 홈과 CRM의 본문 패널은 한 열로 쌓음
- 요약 카드는 두 열을 기본으로 사용
- 개별 기능 카드만 사용자의 요청에 따라 네 열을 유지
- 가로 스크롤이 필요한 CRM 보드는 스냅 가능한 가로 영역으로 제공

## 5. 공통 디자인 시스템

### 색상

```css
--command-bg: #F0F4F2;
--command-surface: #FFFFFF;
--command-text: #17312E;
--command-muted: #728680;
--command-teal: #155855;
--command-teal-strong: #123F3B;
--command-teal-soft: #EAF3F1;
--command-orange: #E85D2C;
--command-amber: #EB8F22;
--command-success: #22876A;
--command-danger: #DC4C4C;
```

티얼과 중립색이 화면의 약 70%, 흰색 카드가 20%, 오렌지 및 상태색이 10% 이내가 되도록 한다. 오렌지를 장식 목적으로 반복 사용하지 않는다.

### 타이포그래피

- 기본: `Pretendard Variable`
- 한국어 시스템 대체: `Apple SD Gothic Neo`, `Noto Sans KR`, sans-serif
- 페이지 제목: 28~34px, 800 이상
- 섹션 제목: 16~18px, 800 이상
- 카드 제목: 12~14px, 750 이상
- 본문: 10~12px
- 상태 및 eyebrow: 8~10px, 대문자 또는 굵은 한글
- 숫자: 고정폭에 가까운 숫자 형태와 `font-variant-numeric: tabular-nums`

새 웹폰트 로딩이 성능이나 배포 안정성을 해치면 기존 로컬 폰트 스택을 유지하고 크기·굵기·자간만 적용한다.

### 카드

- 일반 카드 곡률: 16~18px
- 큰 패널 곡률: 20px
- 주요 카드에는 3px 외곽 셸과 내부 코어를 사용하는 double-bezel 구조 적용
- 외곽선은 진한 회색 실선 대신 티얼 투명도 5~8%의 링 사용
- 그림자는 `rgba(21, 65, 59, 0.05~0.10)` 범위의 확산형 그림자만 사용
- hover 시 최대 2px 상승하며 카드 전체 확대는 사용하지 않음

### 버튼과 상태 배지

- 주요 버튼은 pill 형태
- 주요 버튼의 화살표 또는 추가 아이콘은 독립된 원형 영역에 배치
- 상태 배지는 텍스트와 색상을 함께 사용
- 상태 체계: 진행 중, 확인 필요, 지연, 대기, 완료
- 카드 전체가 링크일 때 내부에 중복 링크를 만들지 않음

### 모션

- 기본 전환: `550ms cubic-bezier(.22, 1, .36, 1)`
- 애니메이션 속성은 `transform`과 `opacity`만 사용
- 카드 hover 이동은 `translateY(-2px)` 이내
- 운영 화면이므로 큰 진입 애니메이션과 반복 애니메이션은 사용하지 않음
- `prefers-reduced-motion`에서는 전환을 사실상 제거

## 6. 관리자 홈

정보 순서는 다음과 같다.

1. 오늘 브리핑과 오늘의 명언
2. 핵심 KPI 요약 카드
3. 오늘의 촬영 일정
4. 오늘 처리할 업무
5. 올리비아 추천 액션
6. 최근 활동과 빠른 실행

### 세부 규칙

- 브리핑은 딥 티얼 배경으로 가장 먼저 인식되게 함
- 명언은 밝은 중립 카드로 브리핑과 대비
- KPI는 데스크톱에서 한 줄 비교가 가능하도록 압축
- 일정과 업무 목록은 테이블과 카드의 중간 형태로 구성
- 메인 목록은 넓게, 추천·최근 활동·빠른 실행은 오른쪽 레일에 배치
- 예시 데이터는 실제 데이터처럼 오해되지 않도록 기존 프레젠테이션 값 안내 원칙 유지

## 7. 고객관리 CRM

정보 순서는 다음과 같다.

1. 프로젝트 상태 KPI
2. 4단계 프로젝트 보드
3. 최근 고객 및 다음 액션
4. 지연·확인 필요
5. 올리비아 추천 액션

### 세부 규칙

- KPI는 견적, 계약, 촬영, 셀렉, RAW, 보정, 후속 관리 상태를 빠르게 비교할 수 있게 함
- 프로젝트 보드는 상담·견적, 촬영 준비, 셀렉·보정, 납품·후속의 네 열 유지
- 보드 카드는 이름, 프로젝트 유형, 현재 단계만 노출
- 지연 항목은 위험색을 카드 전체에 칠하지 않고 배지와 왼쪽 표시선으로 구분
- 모바일 보드는 열을 세로로 눌러 담지 않고 가로 스크롤 사용

## 8. 개별 기능

### 데스크톱

- 1440px 이상: 5열
- 768~1439px: 4열
- 모든 카드는 `aspect-ratio: 1`
- 카드 구성: 아이콘, 기능명, 한 줄 설명, 분류 태그, 상태, 열기 액션
- 상단에는 고객 연결 배너와 검색·카테고리 필터를 배치

### 모바일

- 화면 폭과 관계없이 4열 유지
- 카드 크기가 작아지는 것을 허용
- 설명, 태그, 상태, 별도 열기 버튼은 숨김
- 아이콘과 최대 두 줄 기능명만 표시
- 카드 전체를 단일 링크로 사용
- 기능명은 중앙 정렬하고 두 줄 이후 말줄임
- 카드 사이 간격은 5~6px
- 작은 화면에서도 카드 간 구분이 보이도록 배경과 외곽 링의 명도 차이를 유지

## 9. 데이터 상태

모든 데이터 영역은 다음 네 상태를 구분한다.

### 정상

- 실제 수치와 마지막 갱신 시각 표시
- 재조회 중에도 기존 데이터를 유지

### 로딩

- 최종 레이아웃과 동일한 크기의 스켈레톤 사용
- 로딩 과정에서 카드 높이와 페이지 구조가 흔들리지 않게 함

### 빈 데이터

- 정상적으로 항목이 없다는 문구 표시
- 일정 추가, 고객 등록, 기능 실행 등 다음 행동 제공

### 연결 오류

- 빈 데이터 문구를 사용하지 않음
- 오류 설명과 다시 불러오기 제공
- 다른 패널의 데이터가 정상이라면 페이지 전체를 오류 화면으로 교체하지 않음
- 마지막 정상 데이터가 있으면 유지하고 연결 상태만 표시

로딩과 오류 메시지는 `aria-live`로 알리며 색상만으로 상태를 전달하지 않는다.

## 10. 컴포넌트 변경 경계

- `AdminShell`: 공통 224px/72px 레이아웃과 모바일 드로어 관리
- `AdminSidebar`: 고밀도 메뉴 간격, 현재 위치, 스크롤, 모바일 포커스 관리
- `AdminHeader`: 압축된 검색·알림·프로필·로그아웃 구조
- `SummaryCard`: compact KPI와 상태별 아이콘 톤
- `CategorySection`: 공통 패널 셸과 헤더 간격
- `ActionCard`: 목록형 업무 카드
- `OliviaRecommendationPanel`: 딥 티얼 추천 레일
- `FeatureCard`: 데스크톱 상세형과 모바일 축약형 스퀘어 카드
- 신규 상태 컴포넌트가 필요하면 `DataStatePanel` 또는 동등한 단일 컴포넌트로 분리

페이지는 데이터를 준비하고 컴포넌트에 전달하는 역할만 맡는다. 시각 상태와 반응형 표현은 공통 컴포넌트가 담당한다.

## 11. 성능과 접근성

- 공통 CSS는 기존 `admin.css` 범위에서 토큰과 컴포넌트 스타일을 관리
- 불필요한 클라이언트 컴포넌트 추가 금지
- 무거운 애니메이션 라이브러리 추가 금지
- 브라우저 너비 감지는 CSS 우선, 상호작용에만 `matchMedia` 사용
- 모든 아이콘 버튼에 접근 가능한 이름 제공
- 키보드 포커스 표시 유지
- 모바일 사이드바 포커스 트랩과 Escape 닫기 지원
- 최소 터치 영역 원칙은 일반 액션에 유지하되, 사용자가 승인한 모바일 4열 기능 카드는 카드 전체를 터치 영역으로 사용

## 12. 검증 기준

### 정적 검증

- `npx tsc --noEmit --incremental false`
- `npm run build`
- `git diff --check`

### 브라우저 검증

- 1440px 데스크톱
- 1024px 노트북 또는 태블릿 가로
- 390px 모바일
- 320px 소형 모바일

### 확인 항목

- 관리자 홈, CRM, 개별 기능이 동일한 디자인 시스템을 사용함
- 사이드바가 데스크톱과 모바일에서 정상 동작함
- 개별 기능 카드가 데스크톱 5/4열, 모바일 4열로 표시됨
- 모바일 기능명이 두 줄을 넘지 않음
- 카드 전체 링크에 중첩 링크가 없음
- 로딩, 빈 데이터, 연결 오류가 서로 다른 문구로 표시됨
- 한 API의 실패가 다른 정상 패널을 가리지 않음
- 키보드 탐색과 포커스 이동이 정상임
- `prefers-reduced-motion`에서 불필요한 애니메이션이 제거됨
- 브라우저 콘솔에 새 오류가 없음

## 13. 완료 기준

- 세 관리자 화면이 Dense Command 방향으로 일관되게 보인다.
- 대표가 첫 화면에서 KPI, 일정, 지연 상태, 다음 행동을 빠르게 판단할 수 있다.
- 개별 기능은 데스크톱 스퀘어 그리드와 모바일 4열 스퀘어 그리드로 표시된다.
- 디자인 변경으로 기존 기능 링크와 고객 연결 파라미터가 손실되지 않는다.
- 데이터 오류가 빈 상태로 오인되지 않는다.
- 타입 검사, 프로덕션 빌드, 지정 뷰포트의 실제 브라우저 검증을 통과한다.
