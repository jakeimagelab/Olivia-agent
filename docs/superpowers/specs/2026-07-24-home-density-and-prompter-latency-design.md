# 홈 화면 밀도 및 프롬프터 미러링 지연 개선 설계

## 목적

홈 화면을 1440×900에서 빠르게 파악할 수 있는 밀도로 줄이고, 같은 네트워크에 있는 PC와 태블릿의 프롬프터 미러링을 더 즉각적이고 부드럽게 만든다. 기존 데이터, 인증, Supabase Realtime 연결은 보존한다.

## 홈 화면

### 빠른 실행

- 어시스턴트 홈의 `admin-menu-card` 시각 언어를 축소해 사용한다.
- 캘린더, 고객 등록, 메일링, 고객관리, 사진작업실, 프롬프터를 데스크톱 한 줄 6열로 배치한다.
- 6개 모두 아이콘, 영문 메타, 기능명, 짧은 설명을 사용하되 설명은 한 줄로 제한한다.
- 태블릿은 3열 또는 6열, 모바일은 3×2로 배치하고 가로 스크롤을 만들지 않는다.

### 상단 카드

- 기존 3개 카드의 구조와 높이는 유지한다.
- 가운데 TODAY SCHEDULE의 일정 시간과 제목을 현재보다 크게 표시한다.
- 긴 일정 제목은 한 줄 말줄임 처리해 카드 높이를 늘리지 않는다.

### 업무·마케팅 브리핑

- 관리자 UI와 같은 글꼴 상속을 강제하고 별도 폰트 인상을 제거한다.
- 패널 패딩, 필터, 항목 간격, 버튼 높이를 줄인다.
- 홈에서는 각 브리핑의 우선 항목 3개만 보여주고 전체 목록은 기존 상세 화면으로 연결한다.
- 업무 항목은 상태·제목·고객·기한·실행만 한 행 중심으로 표시한다.
- 마케팅 항목은 카테고리·제목·두 줄 요약·행동 버튼을 표시하며 추천 행동 박스의 중첩 카드를 제거한다.
- 1440×900에서 상단 3개 카드, 빠른 실행, 두 브리핑의 핵심 항목이 첫 화면에 보이는 것을 목표로 한다.

## 홈 데이터 로딩

- `/api/dashboard` 요청은 홈 단위에서 한 번만 실행하고 상단 브리핑, 오늘 스케줄, 업무 브리핑이 같은 결과를 공유한다.
- `/api/dashboard`, `/api/workflow/summary`, 마케팅 데이터 요청은 서로 독립적인 범위에서 병렬 실행한다.
- 이전 데이터를 유지하면서 백그라운드 갱신해 전체 패널을 로딩 화면으로 되돌리지 않는다.
- 동일 요청이 진행 중이면 기존 Promise를 재사용해 중복 호출을 막는다.
- 캘린더 완료 이벤트는 관련 데이터만 갱신한다.

## 승인·확인 반응성

- 사용자가 승인, 확인, 완료를 누르면 해당 항목 상태와 버튼을 즉시 변경한다.
- 처리 중인 항목만 disabled 처리하고 페이지 전체 로딩은 표시하지 않는다.
- 서버 성공 후 최신 데이터와 조정하고, 실패하면 이전 상태로 복구하며 인라인 오류를 표시한다.
- 서버는 기존 권한과 상태 전이 검증을 계속 담당하며 클라이언트의 낙관적 상태를 신뢰하지 않는다.

## 프롬프터 직접 연결

### 연결 구조

- 기존 Supabase Broadcast 채널은 연결 신호 교환과 fallback 용도로 유지한다.
- PC가 WebRTC offer를 생성하고 태블릿이 answer를 반환한다.
- ICE candidate는 기존 Supabase 채널로 교환한다.
- 직접 연결 성공 시 고빈도 스크롤 프레임과 양방향 조작 명령은 WebRTC DataChannel을 우선 사용한다.
- 직접 연결이 실패하거나 끊기면 Supabase Broadcast로 자동 복귀한다.
- 재연결 성공 시 사용자 조작 없이 직접 연결로 복귀한다.

### 채널

- `control`: ordered, reliable. 재생, 정지, 문단 이동, 설정 변경에 사용한다.
- `motion`: unordered, 재전송 없음. 최신 스크롤 프레임만 의미가 있으므로 오래된 패킷은 폐기한다.
- 모든 메시지는 기존 sequence 번호 검증을 유지해 역순 도착을 무시한다.

### 부드러운 미러링

- PC는 매 프레임 React state를 갱신하지 않고 DOM ref로 자체 진행률을 표시한다.
- 네트워크 전송은 약 20~30Hz로 제한하고 위치·전송 시각·재생 상태를 보낸다.
- 태블릿은 수신 위치를 목표값으로 저장하고 `requestAnimationFrame`에서 60fps로 보간한다.
- 재생 중에는 속도를 예측해 앞으로 진행하며 다음 프레임에서 작은 오차만 보정한다.
- 재생/정지, 재시작, 문단 이동, 직접 seek는 즉시 적용한다.
- 태블릿 사용자가 직접 스크롤하는 동안에는 기존 interaction lease를 유지해 PC 위치가 덮어쓰지 않게 한다.
- `scrollProgress`의 고빈도 React state 갱신을 제거하고 스크롤과 진행 바를 ref로 갱신한다.

## 실패 처리와 호환성

- WebRTC를 지원하지 않는 브라우저는 기존 Supabase 방식만 사용한다.
- 연결 상태는 `직접 연결`, `클라우드 연결`, `재연결 중`으로 구분하되 화면을 복잡하게 만들지 않는다.
- 직접 연결 실패는 재생을 막지 않는다.
- iPad Safari와 Android Chrome의 기본 WebRTC API만 사용하며 새 패키지는 추가하지 않는다.

## 검증

- 1440×900에서 홈 핵심 영역이 첫 화면에 표시되는지 확인한다.
- 390px 모바일에서 빠른 실행 3×2, 브리핑 1열, 가로 스크롤 없음 여부를 확인한다.
- 홈 진입 시 `/api/dashboard` 호출 횟수가 한 번인지 확인한다.
- 일정 완료 후 상단 카드와 업무 브리핑이 함께 갱신되는지 확인한다.
- PC→태블릿, 태블릿→PC 재생·정지·seek를 양방향 확인한다.
- WebRTC 연결 성공, 강제 실패 후 Supabase fallback, 재연결을 확인한다.
- 미러링 자동 스크롤 중 불필요한 전체 React 렌더링이 발생하지 않는지 확인한다.
- `npm run typecheck`, `npm run test`, `npm run build`를 통과한다.

## 제외 범위

- 영상 스트리밍 방식의 화면 공유는 추가하지 않는다.
- 기존 프롬프터 UI, 대본 편집 구조, Supabase 테이블은 변경하지 않는다.
- 새로운 서버나 DB 마이그레이션을 추가하지 않는다.
