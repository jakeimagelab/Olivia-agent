# 올리비아 채팅 중복 표시 방지 설계

## 원인

운영 DB에는 메시지가 한 번만 저장된다. 다만 `olivia_chat_messages.metadata`가 없는 환경에서는 현재 탭을 식별하는 `deviceId`와 요청을 식별하는 `clientRequestId`가 저장되지 않는다. 화면에 즉시 표시한 메시지가 DB 폴링으로 다시 도착하면 별개 메시지로 합쳐져 중복 표시된다.

## 해결

- 클라이언트에서 메시지를 화면에 추가하기 전에 `clientRequestId`를 생성한다.
- 메시지 저장 API는 삽입된 행의 `id`와 `created_at`을 반환한다.
- 저장 응답이 오면 낙관적으로 표시한 메시지에 DB ID를 연결한다.
- 폴링이 먼저 도착하는 경쟁 상황에서는 DB ID가 없는 낙관적 메시지 중 역할·내용·출처가 같고 생성 시각이 가까운 한 건만 교체한다.
- DB ID가 있거나 요청 ID가 같은 메시지는 기존처럼 확정적으로 중복 제거한다.
- 사용자가 같은 문장을 나중에 다시 전송하는 정상 사용은 유지한다.

## 호환성

- `metadata` 컬럼이 있는 DB에서는 요청 ID와 기기 ID를 사용한다.
- `metadata` 컬럼이 없는 DB에서는 서버가 반환한 DB ID와 제한된 낙관적 메시지 교체를 사용한다.
- metadata 컬럼 추가용 멱등 마이그레이션 파일을 함께 둔다.

## 검증

- 동일한 저장 메시지가 폴링되어도 한 번만 표시되는지 테스트한다.
- 같은 문장을 서로 다른 시점에 보낸 경우 두 건이 유지되는지 테스트한다.
- 타입 검사와 프로덕션 빌드를 실행한다.
- Git 커밋, 푸시, Vercel 배포는 수행하지 않는다.
