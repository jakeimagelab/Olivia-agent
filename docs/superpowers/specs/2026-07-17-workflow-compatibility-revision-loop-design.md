# Olivia 12단계 워크플로우 호환 전환 및 고객 수정요청 루프

## 1. 범위

이 문서는 전체 개편 요청 중 첫 번째 하위 작업만 다룬다.

- 기존 18단계 워크플로우와 진행 중 프로젝트를 보존한다.
- 신규 프로젝트의 실제 실행 흐름을 4스테이지 12단계로 전환한다.
- 기존 흡수 단계에 머물러 있는 프로젝트가 데이터 손실 없이 신규 흐름에 합류하도록 한다.
- 고객이 승인된 결과물에 수정을 요청하고, 대표가 재작업 후 다시 승인할 수 있는 반복 루프를 완성한다.

관리자·고객 화면의 대규모 시각 개편, 리워드 적립, 데이터 만료 알림, `_archived` 이동은 후속 하위 작업에서 다룬다.

## 2. 설계 원칙

1. 기존 단계 키와 DB 행을 삭제하거나 일괄 치환하지 않는다.
2. 단계 정의 카탈로그와 신규 실행 순서를 분리한다.
3. 신규 실행은 12단계만 사용하고, 흡수된 단계는 부모 단계의 내부 작업으로 실행한다.
4. 기존 실행이 흡수 단계에 있으면 명시적인 호환 전환표를 사용한다.
5. 고객 입력값으로 다른 고객·프로젝트·워크플로우를 수정할 수 없도록 포털 세션을 기준으로 검증한다.
6. 수정 요청, 대표 알림 작업, 승인 상태 변경은 하나의 트랜잭션으로 처리한다.

## 3. 신규 4스테이지 12단계

| 표시 순서 | 실행 키 | 표시 이름 | 스테이지 |
|---:|---|---|---|
| 1 | `consult_meeting` | 문의접수 / 신규등록 | 상담·계약 |
| 2 | `quote` | 방문상담 / 견적생성 | 상담·계약 |
| 3 | `contract` | 견적컨펌 / 계약생성 | 상담·계약 |
| 4 | `conti` | 콘티생성 | 준비·촬영 |
| 5 | `shooting` | 촬영 | 준비·촬영 |
| 6 | `payment_confirm` | 잔금 / 계산서 | 준비·촬영 |
| 7 | `backup_sorting` | 백업 | 데이터·공유 |
| 8 | `client_selection` | 사진분류 / 셀렉 | 데이터·공유 |
| 9 | `retouching` | 보정 | 데이터·공유 |
| 10 | `final_delivery` | 보정본전달 | 피드백·완료 |
| 11 | `revision` | 피드백수렴 | 피드백·완료 |
| 12 | `reward` | 최종완료 / 리워드 | 피드백·완료 |

`payment_confirm`은 기존 코드의 `order_index: 18`을 실행 순서로 사용하지 않는다. 신규 실행 순서는 별도 상수의 배열 위치로 결정한다.

## 4. 단계 모델

`lib/workflow.ts`는 다음 역할을 분리한다.

### 4.1 전체 단계 카탈로그

기존 `WORKFLOW_STEPS`는 레거시 키를 포함한 전체 정의를 보관한다. DB에 저장된 기존 키 해석과 기존 프로젝트 완료를 위해 삭제하지 않는다.

### 4.2 활성 실행 단계

`ACTIVE_WORKFLOW_STEP_KEYS`와 `ACTIVE_WORKFLOW_STEPS`를 추가한다. 신규 프로젝트 시작, 진행률, 다음 단계 계산, 12단계 타임라인은 이 목록만 사용한다.

### 4.3 내부 흡수 단계

`INTERNAL_STEP_GROUPS`를 추가한다.

- `client_selection`: `original_delivery`, `raw_matching`
- `final_delivery`: `seo_delivery`
- `revision`: `review_content`

신규 실행에서는 흡수 키를 `workflow_runs.current_step_key`에 기록하지 않는다. 기존 자동 작업은 부모 단계의 `agent_tasks.task_type`으로 생성하고 `workflow_step_key`는 부모 키를 사용한다.

### 4.4 상시 기능

`customer_care`, `content_planning`은 `STANDALONE_WORKFLOW_STEP_KEYS`로 분리한다. 메뉴와 독립 기능은 유지하되 신규 워크플로우 다음 단계 계산에서 제외한다.

## 5. 기존 프로젝트 호환

기존 프로젝트의 `current_step_key`를 강제로 일괄 수정하지 않는다. 다음 단계로 진행할 때 아래 전환표를 적용한다.

| 기존 현재 키 | 다음 실행 키 |
|---|---|
| `original_delivery` | `client_selection` |
| `raw_matching` | `retouching` |
| `seo_delivery` | `final_delivery` |
| `review_content` | `reward` |
| `customer_care` | 완료 |
| `content_planning` | 완료 |

화면 진행률과 스테이지 표시는 기존 흡수 키를 부모 키로 정규화해 계산한다. 기존 단계의 미완료 작업과 승인 기록은 그대로 유지하며, 해당 단계가 완료된 뒤에만 전환표를 적용한다.

## 6. 자동 진행 변경

- `getNextWorkflowStep`은 `order_index` 정렬 대신 활성 12단계 순서를 사용한다.
- 레거시 흡수 키는 5장의 전환표를 우선 적용한다.
- `buildWorkflowNextAction`의 진행률과 마지막 단계 판정은 활성 12단계를 기준으로 계산한다.
- `createStepTasks`는 활성 부모 단계에 흡수된 내부 작업 정의를 함께 생성한다.
- 동일 단계 재실행 시 기존 고유 인덱스와 충돌하지 않도록 일반 자동 작업은 멱등성을 유지한다.
- 신규 워크플로우는 `consult_meeting`에서 시작하며 `reward` 완료 후 워크플로우를 완료한다.

## 7. 고객 수정요청 데이터 모델

기존 `client_revision_requests`를 확장하는 SQL 마이그레이션을 추가한다.

- `project_id`
- `workflow_run_id`
- `step_key`
- `approval_id`
- 수정 요청 본문은 기존 `content` 컬럼을 그대로 사용하고 별도 메시지 컬럼은 만들지 않음
- 반복 요청을 구분할 수 있는 생성 시각과 기존 상태 이력 유지

고객 수정 상태는 `workflow_step_runs`의 제한된 상태 체크를 변경하지 않는다. 다음 데이터로 파생한다.

- 수정 요청 본문과 이력: `client_revision_requests`
- 대표 승인 상태: `agent_approvals.status = revision_requested`
- 대표 작업 알림: `agent_tasks`
- 감사 기록: `agent_logs`

## 8. 수정요청 API

새 중복 엔드포인트를 만들지 않고 기존 `POST /api/client-portal/revision`을 하위 호환 방식으로 확장한다.

### 요청

기존 요청 형식을 계속 허용하고 다음 필드를 추가로 받는다.

```json
{
  "projectId": "optional-project-id",
  "stepKey": "final_delivery",
  "approvalId": "approved-approval-id",
  "message": "수정 요청 내용"
}
```

### 검증

1. `x-portal-token`으로 포털 세션을 확인한다.
2. 요청한 워크플로우와 프로젝트가 세션의 고객에게 속하는지 서버에서 확인한다.
3. 요청 단계가 고객에게 공개 가능한 단계인지 확인한다.
4. 대상 승인 건이 실제로 `approved` 상태인지 확인한다.
5. 요청 메시지가 비어 있으면 400을 반환한다.

### 원자적 처리

Postgres RPC를 사용해 다음 작업을 한 트랜잭션으로 처리한다.

1. `client_revision_requests` 이력 생성
2. 대상 `agent_approvals`를 `revision_requested`로 변경
3. 고유한 수정 검토 `agent_tasks` 생성
4. `agent_logs` 기록 생성

중간 단계가 실패하면 전체 요청을 롤백한다. 클라이언트에는 내부 DB 메시지를 노출하지 않고 재시도 가능한 오류를 반환한다.

## 9. 재작업 및 재승인

대표가 수정 작업을 완료한 뒤 같은 승인 건을 다시 제출할 수 있도록 재제출 엔드포인트를 추가한다.

```text
POST /api/agent/approvals/[id]/resubmit
```

- 현재 상태가 `revision_requested`인지 확인한다.
- 연결된 수정 검토 작업을 완료 처리한다.
- 승인 상태를 `pending`으로 되돌리고 이전 승인 시각은 감사 로그에 남긴다.
- 고객 화면에는 다시 승인되기 전까지 해당 결과물을 `수정 진행 중`으로 표시한다.
- 대표가 기존 승인 API로 다시 승인하면 고객에게 재공개한다.
- 같은 승인 건에 대해 여러 차례 수정 요청과 재승인이 가능해야 한다.

## 10. 공개 규칙

- 고객 포털은 `visible_to_client`만으로 결과물을 공개하지 않는다.
- 승인 대상 단계는 연결된 `agent_approvals.status === approved`인 결과물만 공개한다.
- `revision_requested` 또는 재제출 후 `pending` 상태에서는 기존 승인 결과물을 `수정 진행 중`으로 표시하고 다운로드·확정 액션을 제한한다.
- 승인 대상이 아닌 공개 단계는 기존 포털 이벤트와 단계 완료 상태를 사용한다.

이 공개 규칙의 전체 UI 적용은 고객 포털 하위 작업에서 수행하지만, API 응답에 필요한 상태 필드는 이번 작업에서 준비한다.

## 11. 오류 처리와 관측성

- 유효하지 않은 포털 세션은 401, 다른 고객 데이터에 대한 접근은 403으로 구분한다.
- 존재하지 않는 단계·승인 건은 404로 처리한다.
- 중복 클릭은 동일 승인 건의 `requested` 또는 `in_progress` 수정 요청을 RPC에서 잠금 조회해 하나의 활성 요청만 생성한다.
- 모든 단계 전환, 수정 요청, 재제출, 재승인은 `agent_logs`에 기록한다.
- 기존 코드의 존재하지 않는 `revision_manage` 단계 이동은 제거한다.

## 12. 검증

### 정적 검증

- TypeScript 오류 0
- `npm run build` 통과
- 신규 12단계 순서와 스테이지 매핑 단위 검증
- 레거시 전환표 단위 검증

### API 검증

- 다른 고객의 승인 건 수정 요청 차단
- 승인 전 결과물에 대한 고객 수정 요청 차단
- 정상 수정 요청 시 네 종류의 관련 데이터가 함께 생성·변경됨
- RPC 실패 시 부분 데이터가 남지 않음
- 수정 요청 → 재제출 → 재승인 반복 가능

### 회귀 검증

- 기존 18단계 프로젝트의 현재 키를 계속 읽을 수 있음
- 기존 흡수 단계의 미완료 작업과 승인 처리가 가능함
- 신규 프로젝트는 12단계 외의 키로 자동 전진하지 않음
- `customer_care`, `content_planning` 독립 메뉴와 직접 실행 경로가 유지됨

## 13. 완료 기준

1. 신규 워크플로우가 정확히 12단계로 실행된다.
2. 잔금 / 계산서 단계가 촬영 다음에 실행된다.
3. 기존 진행 중 프로젝트가 데이터 손실 없이 완료 가능하다.
4. 고객 수정 요청이 대표 작업과 승인 상태에 연결된다.
5. 재작업 후 같은 결과물을 다시 승인하고 공개할 수 있다.
6. 존재하지 않는 `revision_manage` 키가 더 이상 저장되지 않는다.
7. 타입 검사와 프로덕션 빌드가 통과한다.
