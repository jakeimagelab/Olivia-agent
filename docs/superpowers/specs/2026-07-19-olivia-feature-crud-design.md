# Olivia 기능별 생성·수정 도구 1차 설계

작성일: 2026-07-19  
대상: Olivia Agent / Next.js 15 / Supabase PostgreSQL

## 1. 목표

올리비아 채팅에서 사용자가 자연어로 요청한 앱 데이터를 안전하게 생성하거나 수정한다. 기존 페이지와 API, 워크플로우 자동화, 승인 UI를 유지하고 재사용한다. AI가 임의 SQL이나 임의 테이블 조작을 수행하게 만들지 않는다.

대표 사용 예시는 다음과 같다.

- "오블리브의원 고객 등록해줘. 담당자는 김유진이고 피부과야."
- "오블리브의원 담당자 번호를 010-1234-5678로 바꿔줘."
- "다음 주 화요일 오후 2시에 오블리브의원 촬영 일정 등록해줘."
- "라셀의원 견적 초안을 만들고 메모에 의료진 3명이라고 기록해줘."
- "오늘 상담 내용을 일반 메모로 저장해줘."

### 제외 범위

- 데이터 영구 삭제
- 임의 SQL 실행
- 스키마에 등록되지 않은 테이블/컬럼 쓰기
- 이메일, 카카오톡, SMS 자동 발송
- 가격·계약 조건의 무승인 변경
- 외부 NAS 파일 삭제 또는 이동

## 3. 구조

### 3.1 기능 레지스트리

신규 `lib/olivia/crud/registry.ts`가 올리비아가 조작할 수 있는 도메인을 명시한다.

각 정의는 다음 정보를 가진다.

```ts
type OliviaCrudDefinition = {
  domain: string;
  operations: Array<"create" | "update">;
  identityFields: string[];
  createFields: Record<string, FieldRule>;
  updateFields: Record<string, FieldRule>;
  permission: "auto" | "review_required" | "owner_only";
};
```

레지스트리는 허용 목록이다. 모델이 전달한 알 수 없는 필드는 버리고, 필수 필드·형식·길이·enum을 서버에서 다시 검증한다.

### 3.2 도메인 서비스

신규 `lib/olivia/crud/services/*`가 기능별 저장을 담당한다. 공통 실행기가 테이블에 범용 insert/update를 호출하지 않고 다음처럼 명시적인 서비스를 호출한다.

- `clientService.ts`
- `workflowService.ts`
- `memoService.ts`
- `calendarService.ts`
- `quoteService.ts`
- `contractService.ts`
- `contiService.ts`
- `galleryService.ts`
- `reviewService.ts`
- `mailDraftService.ts`
- `agentWorkService.ts`

가능하면 기존 API가 쓰는 공통 비즈니스 함수를 추출해 API와 올리비아 서비스가 함께 사용한다. 서버에서 자기 API를 HTTP로 다시 호출하지 않는다.

### 3.3 올리비아 도구

채팅 모델에는 두 개의 통합 도구를 제공한다.

```text
create_feature_record
update_feature_record
```

입력에는 `domain`, 구조화된 `data`, 대상 식별자와 사용자 요청 원문이 포함된다. 도메인별 필드는 서버 레지스트리로 검증한다. 도구 수를 과도하게 늘리지 않으면서도 실행 권한은 도메인별로 분리한다.

### 3.4 대상 해석

수정 요청은 다음 순서로 대상을 찾는다.

1. 명시적인 UUID
2. 최근 대화의 작업 항목 참조
3. 정확한 고객명·견적번호·계약번호 등 자연키
4. 공백을 무시한 이름 검색
5. 부분 일치

후보가 한 개면 계속 진행하고, 여러 개면 실행하지 않은 채 선택 목록을 반환한다. 후보가 없으면 수정 요청을 생성 요청으로 바꾸지 않는다.

## 4. 실행 흐름

```text
사용자 자연어 요청
→ Claude가 기능 도메인과 create/update 입력 구성
→ 서버가 레지스트리 기반 검증
→ 기존 데이터 및 연결 대상 조회
→ 변경 미리보기 카드 생성
→ 사용자 확인
→ 기능별 서비스 실행
→ Olivia 이벤트와 활동 로그 저장
→ 생성/변경 결과 카드 반환
```

조회 도구는 자동 실행한다. 생성과 수정은 기존 `pendingTool` 확인 UI를 사용한다. 메모처럼 되돌리기 쉬운 내부 기록도 1차에서는 일관되게 확인 후 실행하며, 안정성이 확인되면 향후 자동 실행 옵션을 추가할 수 있다.

## 5. 데이터 규칙

### 고객

- 병원명은 생성 시 필수다.
- 동일 병원명 후보가 있으면 중복 생성 전에 경고한다.
- 고객 생성 시 현재 동작처럼 워크플로우를 자동 시작할지는 미리보기에서 명시한다.
- 기존 고객 수정은 값이 실제로 달라지는 필드만 저장한다.

### 프로젝트와 워크플로우

- 12단계 활성 키를 유지한다.
- `current_step_key` 직접 변경보다 기존 단계 전환 함수를 우선한다.
- 수동 `next_action`은 AI가 자동으로 덮어쓰지 않는다.
- 고객 연결이 없으면 프로젝트 생성을 중단하고 고객 선택 또는 신규 고객 생성을 제안한다.

### 메모와 일정

- 날짜와 시간이 없는 "메모해줘"는 `consultation_memos`에 저장한다.
- 명시적인 날짜 또는 시간이 있는 일정 요청만 `calendar_tasks`에 저장한다.
- 수정 시 메모와 일정을 서로 다른 도메인으로 유지한다.

### 견적·계약·콘티

- 금액은 음수를 금지하고 정수로 정규화한다.
- 견적번호 등 자연키가 같으면 새 레코드 대신 수정 후보로 제안한다.
- 가격, 할인, 계약 조건 변경은 `owner_only`로 분류한다.
- 계약 생성은 가능하면 연결된 견적과 고객을 확인한다.

### 갤러리

- 고객명으로 기존 고객을 우선 연결한다.
- 고객이 없을 때만 `고객도 함께 생성` 여부를 미리보기에서 명시한다.
- `photo_galleries` 스키마 캐시 오류가 발생하면 실제 마이그레이션과 코드의 컬럼을 일치시키며, 오류를 숨기기 위한 무분별한 fallback insert는 사용하지 않는다.

### 메일

- `mailing_queue.status`는 실제 제약인 `draft | ready | sent | failed`만 사용한다.
- 생성과 수정은 초안까지만 수행한다.
- 실제 발송은 기존 별도 승인 도구를 사용한다.

## 6. 권한과 감사 로그

| 수준 | 대상 |
| --- | --- |
| `review_required` | 일반 생성·수정, 일정, 메모, 갤러리, 메일 초안 |
| `owner_only` | 가격·할인·계약 조건·결제 관련 변경 |
| 범위 제외 | 삭제, 외부 발송, 파일 삭제 |

성공과 실패 모두 다음을 기록한다.

- 요청 사용자 유형
- 도메인과 작업 종류
- 대상 ID
- 변경 전/후의 비민감 필드
- 실행 결과
- 오류 코드와 안전한 오류 메시지

전화번호·이메일·계약 전문은 로그에 그대로 복제하지 않는다.

## 7. API와 UI

신규 내부 API:

```text
POST /api/olivia/crud/preview
POST /api/olivia/crud/execute
GET  /api/olivia/crud/capabilities
```

`preview`는 저장하지 않고 대상 해석, 검증 결과와 변경 전후를 반환한다. `execute`는 preview에서 발급한 짧은 수명의 실행 토큰과 동일한 정규화 입력을 요구해 미리보기 이후 내용이 바뀌는 문제를 막는다.

채팅 카드에는 다음을 표시한다.

- 기능명과 작업 종류
- 생성할 주요 값 또는 변경 전 → 변경 후
- 연결 고객·프로젝트
- 자동으로 함께 일어나는 동작
- 권한 배지
- `확인하고 실행` / `취소`

## 8. 오류 처리

- 하나의 기능 오류가 채팅 전체 응답을 실패시키지 않는다.
- Supabase 오류 코드를 사용자용 메시지와 개발 로그로 분리한다.
- 테이블 또는 컬럼이 없는 경우 해당 기능만 `사용 불가`로 표시한다.
- 생성 성공 후 로그 기록 실패가 발생해도 생성 결과는 성공으로 반환하되 감사 로그 보완 경고를 남긴다.
- 네트워크 재시도로 중복 생성되지 않도록 요청별 idempotency key를 사용한다.

## 9. 테스트 기준

### 단위 테스트

- 허용되지 않은 필드 제거
- 필수 필드와 enum 검증
- 동명 고객 후보 처리
- 메모와 일정 라우팅 구분
- 금액·날짜·연락처 정규화
- 수동 next action 보호
- mailing status 제약 준수
- idempotency 중복 방지

### 통합 테스트

1. 고객 생성 → 워크플로우 연결 → 이벤트/로그 확인
2. 고객명 수정 → 변경 전후 카드 → 단일 레코드 수정
3. 일반 메모 요청 → 캘린더가 아닌 메모 저장
4. 날짜가 있는 일정 요청 → 캘린더 저장
5. 기존 견적 수정 → 새 견적 중복 생성 없음
6. 갤러리 생성 → 기존 고객 자동 연결
7. 메일 초안 생성 → `draft` 상태 저장, 자동 발송 없음
8. 가격 변경 → 대표 전용 승인 없이는 실행 차단

최종적으로 `npm run typecheck`, 관련 Vitest, `npm run build`가 성공해야 한다.

## 10. 완료 기준

- 올리비아가 지원 도메인의 생성·수정 요청을 구조화한다.
- 실행 전에 정확한 변경 내용을 보여준다.
- 모호한 대상을 임의로 수정하지 않는다.
- 기존 API·워크플로우·승인 동작을 깨뜨리지 않는다.
- 모든 작업이 허용 필드 검증과 권한 검사를 통과한다.
- 메모 요청이 캘린더로 잘못 저장되지 않는다.
- 갤러리가 기존 고객과 연결된다.
- 메일 초안은 허용된 status로 저장되고 자동 발송되지 않는다.
- 중복 실행과 중복 생성이 방지된다.
- 타입 검사와 빌드가 성공한다.
