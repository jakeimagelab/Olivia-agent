## 🟡 부분 진행 — 4번 항목 (배경컬러 통일)

**핵심 발견 (중요, 재조사 불필요)**:
메일링 페이지의 실제 렌더링 배경은 평평한 단색이 아니라 **mesh 그라디언트**
(`#f0f4f2` 베이스 + teal/orange 방사형 그라디언트 3겹)임. Playwright로 실제 렌더링 확인해서
검증함 (grep으로 소스 코드만 보고 판단하면 틀림 — 아래 이유).

`app/globals.css`에 이런 규칙이 있음:
```css
:root { --mesh-bg: radial-gradient(...), radial-gradient(...), radial-gradient(...), #f0f4f2; }
main:has(> .pc-header),
div:has(> .pc-header) { background: var(--mesh-bg) !important; }
```
`<PageHeader>` 컴포넌트를 `<main>`/`<div>` 안쪽 **첫 자식**으로 두는 페이지는 이 규칙 때문에
자기가 지정한 배경색과 무관하게 **자동으로** mesh-bg를 받음. 그래서 `daily-ideas`,
`brand-analysis`, `diagnosis`, `clients`, `channel-analyzer` 같은 페이지들은 로컬 테마 색이
제각각이어도(`#EDF5F3`, `#faf7f2`, `#F0F9F8` 등) **실제로는 이미 mailing과 동일하게
보이고 있었음** — 손댈 필요 없음.

**실제로 안 맞았던 것 & 고친 것**:
- `app/photoclinic/page.tsx` (= `/quote` 라우트): `<PageHeader>`를 `<main>` **바깥** 형제로
  둬서 저 자동 규칙을 못 받고 자체 `#faf7f2`(나중엔 실수로 `#F5F0E8`로 잘못 바꿨다가 재수정)를
  쓰고 있었음 → `<main>`에 `style={{ background: "var(--mesh-bg)" }}` 직접 적용해서 통일 완료.
  PDF 생성 팝업창 2개(별도 `document.write` HTML이라 CSS 변수 못 씀)의 배경도 `#f0f4f2`로 통일.

**아직 실제 렌더링 미검증 — 다음에 확인해야 할 후보 4곳** (grep 기준으론 다른 색이지만
mesh-bg 자동 규칙을 받는지 안 받는지 Playwright나 브라우저로 직접 확인 후 판단할 것):

## 3. 신규: 최근 활동 로그 위젯

### 3-1. 신규 컴포넌트 `components/admin/RecentActivityWidget.tsx`

챗봇이 처리한 일(`agent_logs`, `olivia_events`)과 대표님이 직접 처리한 일을 구분 없이 최신순으로 섞어서 5~6개만 보여줌:

```tsx
"use client";
import { useEffect, useState } from "react";

type ActivityItem = { id: string; message: string; createdAt: string; source: "olivia" | "manual" };

export default function RecentActivityWidget() {
  const [items, setItems] = useState<ActivityItem[]>([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch("/api/admin/recent-activity?limit=6", { cache: "no-store" })
      .then((r) => r.json())
      .then((json) => setItems(json.items ?? []))
      .finally(() => setLoading(false));
  }, []);

  return (
    <section className="recent-activity-widget">
      <div className="recent-activity-title">최근 활동</div>
      {loading ? (
        <p className="recent-activity-empty">불러오는 중…</p>
      ) : items.length === 0 ? (
        <p className="recent-activity-empty">최근 활동이 없어요.</p>
      ) : (
        <ul>
          {items.map((item) => (
            <li key={item.id}>
              <span className={`dot dot--${item.source}`}/>
              <span className="msg">{item.message}</span>
              <time>{formatRelative(item.createdAt)}</time>
            </li>
          ))}
        </ul>
      )}
    </section>
  );
}

function formatRelative(iso: string) {
  const diffMin = Math.round((Date.now() - new Date(iso).getTime()) / 60000);
  if (diffMin < 1) return "방금";
  if (diffMin < 60) return `${diffMin}분 전`;
  const diffHour = Math.round(diffMin / 60);
  if (diffHour < 24) return `${diffHour}시간 전`;
  return new Date(iso).toLocaleDateString("ko-KR", { month: "numeric", day: "numeric" });
}
```

### 3-2. 신규 API `app/api/admin/recent-activity/route.ts`

`agent_logs`(챗봇/자동화가 만든 로그)를 최신순으로 가져와서 반환. `log_type`에 따라 `source: "olivia"`로 표시(전부 시스템이 기록하는 로그라 대부분 olivia 소스이지만, 추후 대표님이 웹에서 직접 한 액션도 같은 테이블에 기록하도록 확장 가능 — 일단은 agent_logs 하나로 시작):

```ts
import { NextRequest, NextResponse } from "next/server";
import { getSupabaseAdmin } from "@/lib/supabase";

export const dynamic = "force-dynamic";

export async function GET(req: NextRequest) {
  const limit = Number(req.nextUrl.searchParams.get("limit") ?? 6);
  const db = getSupabaseAdmin();
  const { data, error } = await db
    .from("agent_logs")
    .select("id, message, log_type, created_at")
    .order("created_at", { ascending: false })
    .limit(limit);
  if (error) return NextResponse.json({ ok: false, error: error.message }, { status: 500 });
  const items = (data ?? []).map((row) => ({
    id: row.id,
    message: row.message,
    createdAt: row.created_at,
    source: "olivia" as const,
  }));
  return NextResponse.json({ ok: true, items });
}
```

> `agent_logs` 테이블 존재 여부는 `supabase/workflow-agent.sql` 기준으로 이미 있음 (지난번 진단 참고). 컬럼명이 실제 배포 DB와 다르면 맞춰서 조정.
